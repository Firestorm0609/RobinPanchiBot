# Panchi Bot — Unified Cross-Chain Balance ("FOMO-style") Build Plan

## Goal
Deposit once, on any supported chain, and trade any token on any supported
chain — the bot auto-bridges + auto-swaps behind the scenes, like FOMO's
"unified USD balance across chains" model.

## Product direction change (this checkpoint)
Moved to an embedded-wallets-only model, matching FOMO's architecture:
wallets are always generated by the bot, never imported from an existing
key. Users can still export their keys at any time as a backup. This
removed the need for two other features:

- **Manual chain switching (🔗 Chain tab) — retired.** Chain selection is
  now fully automatic: pasting a token contract address resolves and
  switches the active chain to wherever that token has the best liquidity
  (`resolveChainForCA` in `handlers/text.js`), which already existed
  alongside the manual picker. With import gone there's no scenario left
  where a user needs to manually pick a chain before depositing (see new
  Deposit menu below) or before trading (auto-resolved per token). Users
  still see which chain they're on inside a token card / balance view —
  they just don't pick it by hand anymore.
- **Batch Buy / Batch Sell / Batch Fund / Batch Collect — removed.**
  Unused surface area given the simplified, single-wallet-per-user flow
  this model encourages. `handlers/batch.js` is retired and no longer
  imported by `bot.js`.

New in its place: a dedicated **📥 Deposit** menu (`handlers/deposit.js`) —
pick a chain, get the active wallet's deposit address on that chain
immediately, mirroring the familiar "choose a network to deposit from"
pattern from other wallet apps.

## Key finding (confirmed, don't re-research)
Robinhood Chain is supported by LI.FI (via Jumper, Robinhood's own listed
bridging partner). LI.FI supports EVM<->EVM and EVM<->Solana routes, and
covers every chain this bot lists (Ethereum, Base, Arbitrum, BNB, Robinhood,
Solana).

## Current architecture (baseline)
- `chains.js` — registry of 6 chains (5 EVM + Solana), each with its own
  stablecoin (USDC everywhere except Robinhood=USDG). No UI concept of
  "picking" one anymore — `getActiveChain`/`setActiveChain` are driven
  internally by token resolution, not a user-facing menu.
- `wallet.js` — `createWallet()` only. No import path.
- `storage.js` — positions/balances are per-chain. `pending_trades` has a
  full bridge-aware status machine (see Phase 2/4 below), unaffected by
  the UI changes above.
- `trade-core.js` — `performBuyCore` does pre-flight balance checking,
  auto-bridges a shortfall in from another chain, then swaps. `performSellCore`
  stays same-chain-only. Batch-oriented exports (`distributeUsdc`,
  `estimateTransferGasReserve`, `performCollectCore`) are now unused dead
  code left in the file — no handler calls them anymore; safe to strip in a
  future pass if `trade-core.js` gets revisited.
- `bridge.js` — standalone LI.FI REST wrapper (quote, execute, status poll).
- `pollers.js` — `resumeStuckBridges()` / `startBridgeResumePoller()` actually
  resume a bridge-then-swap trade after a crash/restart, not just alert on it.
- `format.js` — `getUnifiedUsdBalance()` / `formatUnifiedBalanceLines()` give
  a single cross-chain USD total (display-only, not spendable as one balance).
- `price.js` — DexScreener + pump.fun (Solana) + Uniswap Trading API fallback,
  all working.
- `handlers/chain.js` and `handlers/batch.js` — retired, no longer imported
  by `bot.js`. Delete the files from the server; nothing references them.

## Progress so far

### Phase 0 — Prerequisites — DONE
- LI.FI REST API confirmed: base URL `https://li.quest/v1`, no API key
  required (200 req/2hr unauthenticated; `LIFI_API_KEY` gets 200 req/min).
  Default integrator fee 25bps, separate from this bot's own
  `AFFILIATE_FEE_BPS`.
- Decided against `@lifi/sdk` (v4 requires viem; this bot signs with raw
  `ethers.Wallet` / Solana `Keypair` everywhere) — calling the REST API
  directly via axios instead, matching the pattern already used for 0x and
  Jupiter.
- Robinhood Chain bridging confirmed live via Jumper/LI.FI. Solana LI.FI
  chain id confirmed: `1151111081099710`.

### Phase 1 — Standalone bridge module — DONE (code)
`bridge.js`: `getBridgeQuote`, `summarizeBridgeQuote`, `executeBridge`
(signs + submits the source-chain leg, EVM or Solana, then polls
`GET /v1/status` until `DONE`/`FAILED`/timeout), `checkBridgeStatus` for
resuming without re-sending anything. A `TIMEOUT` result is not necessarily
a failure — treated as resumable (see Phase 2/4).

**Still worth doing, but no longer a hard blocker:** run one small real-money
bridge (e.g. Base → Robinhood Chain, USDC→USDG) against production if that
hasn't happened yet — the auto-resume path in `pollers.js` has never been
exercised against a real timeout/restart in the wild. Treat as a pre-launch
smoke test, not a phase-gate.

### Phase 2 — Pending-trade state machine extension — DONE
`storage.js`: `pending_trades` has `bridge_hash` / `bridge_from_chain`
columns (idempotent `ALTER TABLE` migrations). Status enum:
`pending -> [bridging -> bridged ->] swapping -> submitted -> confirmed | failed`.
Same-chain trades skip the bridging/bridged stages entirely, unchanged from
before this migration. `getStuckPendingTradesByKind()` splits stuck trades
into `bridgeStuck` vs `swapStuck`. `pollers.js`'s `checkStuckTrades()` sends
a categorized admin alert on restart.

### Phase 3 — Unified balance UX — DONE
`format.js`: `getUnifiedUsdBalance(wallet)` sums every chain's stablecoin
balance into one display-only USD number (a chain that errors gets
`usd: null`, excluded from the total but still shown). `menus.js`'s
`renderTokenCard()` shows a "Unified balance (all chains)" line.
`handlers/token.js`'s `menu_balance` leads with the unified total, active-chain
breakdown kept below it.

### Phase 4 — Trade flow integration — DONE
`trade-core.js`'s `performBuyCore`:
- Checks target-chain stablecoin balance first; same-chain path is
  unaffected (no bridge check runs) when balance is sufficient.
- On shortfall: rejects below `MIN_BRIDGE_USD` ($5, see `config.js`) with a
  clear "fund this chain directly" message; otherwise finds the chain with
  the largest stablecoin balance via `findBridgeSourceChain` and bridges the
  shortfall (+ `BRIDGE_SHORTFALL_BUFFER_PCT` buffer) via `bridgeShortfall()`.
- Uses the Phase 2 `pending_trades` bridging/bridged/swapping stages to
  survive a restart mid-flight.
- `executeBuy` shows both legs to the user via `onProgress` (bridging
  message with fee/ETA from `summarizeBridgeQuote()`, then the swap
  confirmation) instead of one opaque "fetching quote" message.
- Sells stay same-chain-only (`performSellCore` has no bridge logic).
- Restart-recovery is wired end-to-end, not just alerted on:
  `pollers.js`'s `resumeStuckBridges()` / `resumeOneBridgeTrade()` re-checks
  `checkBridgeStatus()` for anything stuck in `bridging`, transitions to
  `bridged`, then calls `performSwapBuy()` to finish the trade — running
  once at bot startup (`bot.js`) and on a recurring poller
  (`startBridgeResumePoller`, interval `BRIDGE_RESUME_POLL_INTERVAL_MS`).

### Phase 5 — Batch features — REMOVED (was: "compatibility DONE")
Originally shipped as "batch features get bridging for free." Batch
Buy/Sell/Fund/Collect have since been removed entirely as unused surface
area (see "Product direction change" above) — this phase no longer applies.
`handlers/batch.js` is retired.

### Phase 6 — Deposit UX + wallet model simplification — DONE
- Removed `importWallet()` from `wallet.js` and the `import_name`/`import_key`
  pending states from `handlers/text.js` — wallets are create-only now,
  matching the embedded-wallet model.
- Added `handlers/deposit.js`: a `menu_deposit` chain picker showing the
  active wallet's deposit address per chain, replacing the old pattern of
  hunting for an address via 🔗 Chain or 💼 Wallets.
- Removed `handlers/chain.js` and its `menu_chain`/`chain_select_*` actions —
  no longer imported by `bot.js`. Chain selection is fully automatic via
  `resolveChainForCA`.
- `menus.js`: `mainMenu()` updated (Deposit replaces Chain; Batch-related
  buttons removed from `walletsMenu()` and `tokenMenu()`).
- `config.js`: `HELP_TEXT`/`WELCOME_TEXT` updated to stop referencing the
  Chain tab and batch features.

## Remaining open items

1. **Solana-side LI.FI leg unproven.** EVM<->EVM bridging has a real-money
   test plan (see Phase 1 note above); a Solana-source or Solana-destination
   bridge has only been asserted from LI.FI's docs/marketing, never actually
   executed. Test one specifically before fully trusting that path — a
   Solana leg failing inside `bridgeShortfall`/`executeBridge` is the one
   untested branch in the whole flow.
2. **Bridge failure/refund handling is still generic.** `pollers.js` and
   `trade-core.js` handle a `FAILED` LI.FI status by telling the user to
   check their source-chain wallet, but there's no tool-specific knowledge
   of where a partial-failure leaves funds for each underlying bridge LI.FI
   might route through.
3. **`MIN_BRIDGE_USD` floor is set but not revisited.** Currently a flat $5.
   Fine for now; revisit if gas/fee conditions on any chain change enough to
   make $5 bridges routinely uneconomical.
4. **No automated test coverage for the bridge path.** `test/` covers
   `crypto.js` and `storage.js` well but nothing exercises
   `bridge.js`/`trade-core.js`'s bridging branch or `pollers.js`'s resume
   logic — only reachable via live network calls today.
5. **Dead code cleanup.** `trade-core.js` still exports batch-only helpers
   (`distributeUsdc`, `estimateTransferGasReserve`, `performCollectCore`,
   `transferSolanaToken`) with no remaining callers. `solana.js`'s
   `importSolanaWallet`/`isValidSolanaPrivateKey` are similarly orphaned
   after the wallet-import removal. Harmless but worth stripping next time
   either file is touched for another reason.

## How to resume from this checkpoint
1. Read this file fully first.
2. Do NOT re-research: whether Robinhood Chain is bridgeable (yes), LI.FI's
   REST API shape/fee/auth requirements (documented above), or whether to
   use `@lifi/sdk` (no — REST direct). Do re-verify anything LI.FI-specific
   that could have changed (fee %, rate limits, SDK status) if picking this
   up much later.
3. Don't reintroduce wallet import, the Chain tab, or batch features without
   a deliberate decision to reverse the "Product direction change" above —
   they were removed on purpose, not left unfinished.
4. The five items under "Remaining open items" are the actual backlog now.
